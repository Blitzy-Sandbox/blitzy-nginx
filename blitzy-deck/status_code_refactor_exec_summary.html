Risks and Mitigations

RISK REGISTER

Known Constraints

Six identified constraints, each with a deterministic mitigation already in place.

# Risk Mitigation
1 Wire-format reason phrase divergence (302, 408, 413, 414, 416, 503) between RFC 9110 and ngx_http_status_lines[] Wire table preserved byte-for-byte; registry carries canonical phrase for new consumers; tracked in decision log
2 error_page parser legacy validation rules (300–599 + 499 rejection) Parser preserved byte-for-byte; out of refactor scope; behavior change forbidden by AAP
3 Upstream backends emitting non-canonical codes (e.g., 999) API bypasses strict validation when r->upstream != NULL; preserves existing pass-through semantics
4 Strict-mode default could break third-party modules Strict mode is OPT-IN via --with-http_status_validation; default builds remain permissive
5 Two divergent NGX_HTTP_LAST_2XX definitions (207 vs 202) Pre-existing behavior preserved; harmonization explicitly out of scope
6 Per-worker memory budget (<1 KB/worker) Static registry; no heap allocation; rfc_section field moved to debug-only parallel array
